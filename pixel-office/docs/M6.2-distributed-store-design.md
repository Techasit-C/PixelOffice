# M6.2 — Distributed Store Design (Cache + Rate Limiter)

Status: DESIGN ONLY — reviewed by solution-architect, awaiting CEO sign-off. No code, no deploy, no provisioning is authorized by this document.
Sprint: 6 / M6.2. Target runtime: Vercel serverless (`vercel.json` → `"framework": "nextjs"`, `"regions": ["iad1"]`).
Scope: migrate the in-memory **rate limiter** (`lib/api/rate-limit.ts`) and the in-memory **caches** (`lib/agents/agents-cache.ts`, `lib/market-data/service.ts`) to a **distributed, serverless-friendly store**, default **Upstash Redis (REST)**, behind an abstraction so route call-sites are (near-)unchanged.

---

## 1. Problem statement

The limiter and caches keep all state in **module-scoped memory**:

- `lib/api/rate-limit.ts:38` — `InMemoryRateLimiter` holds counters in `private readonly windows = new Map<string, WindowState>()`.
- `lib/agents/agents-cache.ts:31` — the roster cache is a single `let entry: CacheEntry | null`.
- `lib/market-data/service.ts:41-42` — `const memPriceCache = new Map<string, CacheEntry>()` and `let memFx`.

On Vercel every request may hit a **different lambda instance**, and instances are created/destroyed on demand. Module memory is therefore **per-instance, non-shared, and reset on cold start**. Both files already document this caveat honestly (`rate-limit.ts:7-12`, `agents-cache.ts:10-12`), so this migration formalizes a known gap rather than fixing a surprise.

Concrete user-visible / security-relevant failure modes:

| Concern | In-memory failure on serverless | Impact |
|---|---|---|
| Rate limit | Effective global limit = `configured × concurrent instances` (`rate-limit.ts:8-9`). A user (or a leaked/abused session token) hitting a scaled-out endpoint gets N× the intended budget. | The 429 shield that protects upstream provider quotas (Finnhub/CoinGecko) and write endpoints is **not a real cap** — the security control is porous. |
| Rate limit | Counters reset on scale-down / cold start. | A blocked abuser is un-blocked the moment a new instance serves them. |
| Price/FX cache | Each instance rebuilds its own copy → provider is hit up to once-per-instance-per-TTL, not once globally. | Under fan-out this multiplies calls to rate-limited providers; the very quota the `PRICE_TTL_MS` shield (`service.ts:25`) exists to protect is undermined. |
| Agents cache | Per-instance rebuild. | Benign (pure FS read), lowest migration priority. |

The rate limiter is the one with a **security** dimension; the caches are correctness/cost. This ordering drives the fail-mode policy in §5.

## 2. Store selection — Upstash Redis (REST), confirmed

**Decision: adopt Upstash Redis over its REST API as the default backing store. No blocker found.**

Why it fits this codebase and Vercel:

- **Connection model — REST, not TCP pool.** Serverless functions are short-lived and horizontally fan out; a classic Redis client holds a persistent TCP connection and, at scale, exhausts the server's connection budget (the exact reason this repo runs Neon through pgBouncer — see `.env.example:5-11`, `DATABASE_URL` "pooled"). Upstash's HTTP/REST client (`@upstash/redis`) is **stateless per call** — no pool to exhaust, no connection to keep warm across cold starts. This is the single most important reason to pick Upstash-REST over "just run Redis."
- **Vercel-native.** First-class Vercel Marketplace integration; env vars are the standard `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN` pair — plain HTTPS + bearer token, matching the app's existing "read a secret from `process.env`, fail closed" convention (`verify-secret.ts:29-36`, `snapshot-batch.ts:46-58`).
- **Purpose-built primitive.** `@upstash/ratelimit` implements atomic sliding/fixed-window limiting server-side via a single Lua-scripted command — no read-modify-write race across instances.
- **Cost / free tier.** Free tier is generous for a single-user personal-scale app; volume math and the risk it turns into a real limit are in §8 and raised as an Open Question in §9.
- **Latency.** REST-per-call adds one HTTPS round trip (single-digit-to-low-tens ms intra-region when the DB is `iad1` and the Upstash DB is co-located `us-east-1`). Acceptable for the poll cadences here (fastest is 10s); named as a tradeoff in §8.
- **Region.** `vercel.json` pins `iad1` (us-east-1). Provision the Upstash DB in the **same region**; do NOT enable global replication (adds cost + write latency, no benefit for a single-region deployment). Multi-region is a non-goal until the app itself is multi-region.
- **Portability / no lock-in.** Upstash speaks the Redis wire/command protocol. The `@upstash/redis` REST client is Upstash-specific, but the commands (`INCR`, `EXPIRE`, `GET`, `SET EX`, `DEL`) and `@upstash/ratelimit`'s algorithms are standard Redis — swappable for any Redis-compatible host by replacing the adapter (see §6). The app never imports the vendor client outside the adapter.

Alternatives considered:

| Option | Why not (for this milestone) |
|---|---|
| Self-hosted Redis / Elasticache with `ioredis` | Persistent TCP pool is hostile to serverless fan-out (connection exhaustion, cold-start reconnect). Ops burden. |
| Vercel KV | Now a thin layer over Upstash anyway; adopting Upstash directly avoids an extra indirection and keeps portability explicit. |
| Postgres (Neon) as a store | DB already pooled and available, but every limiter check becomes a write on the hot poll path → wasteful, and Neon compute cost/row-churn. Wrong tool for ephemeral TTL counters. |
| Keep in-memory | The status quo this milestone exists to fix. |

## 3. Store contract (the abstraction the app codes against)

Two small interfaces, one new module (proposed `lib/store/`), types only — no implementation here.

```ts
// lib/store/types.ts  (PROPOSED — not yet written)

/** Distributed key/value cache with TTL. All ops are async (network-backed). */
export interface DistributedCache {
  /** Return the parsed value, or null on miss / on backend outage (fail-open read). */
  get<T>(key: string): Promise<T | null>;
  /** Store value with a TTL in seconds. Best-effort; never throws to the caller. */
  set<T>(key: string, value: T, ttlSeconds: number): Promise<void>;
  /** Delete a key. Best-effort; never throws. */
  del(key: string): Promise<void>;
}

/** Atomic fixed/sliding-window limiter primitive. */
export interface DistributedRateLimiter {
  /**
   * Atomically record one hit for `key` within a window and report the verdict.
   * Backed by a single server-side atomic op (Upstash @upstash/ratelimit Lua,
   * or INCR + EXPIRE in one pipeline) — no read-modify-write race across instances.
   */
  check(key: string): Promise<RateLimitResult>; // RateLimitResult already exists in lib/api/rate-limit.ts:15
}
```

Note the **`Promise` on `check`** — this is the one unavoidable signature change vs. today's **synchronous** `RateLimiter.check` (`rate-limit.ts:25`). Analysis and mitigation in §4.

### Key schema / namespacing

Prefix every key so cache and limiter never collide and keys are greppable in the Upstash console:

| Domain | Key pattern | TTL | Notes |
|---|---|---|---|
| Rate limit | `rl:<bucket>:<userId>` | window (default 60s) | `bucket` ∈ `write \| providerRead \| agentsRead \| signalsRead`. Mirrors today's key `` `${bucket}:${userId}` `` (`rate-limit.ts:128`) with an `rl:` prefix. `@upstash/ratelimit` manages the window suffix internally. |
| Price cache | `cache:price:<assetId>` | 60s (`PRICE_TTL_MS`, `service.ts:25`) | keyed by `assetId`, matching `memPriceCache` (`service.ts:41`). |
| FX cache | `cache:fx:usd-thb` | 3600s (`FX_TTL_MS`, `service.ts:26`) | single key. |
| Agents roster | `cache:agents:roster` | 30s (`AGENTS_CACHE_TTL_MS`) | single key; lowest priority to migrate. |

**Atomicity requirement (non-negotiable for the limiter):** the increment and the expiry MUST be one atomic server-side operation. Use `@upstash/ratelimit` (recommended — Lua-scripted, one command) or an `INCR` + `EXPIRE`-on-first-hit executed as a single Upstash pipeline/Lua unit. A JS-side `get` then `set` is a race across concurrent instances and is **forbidden** — it reintroduces the very over-count this milestone removes.

## 4. Rate-limit semantics & preserving the call-site contract

### Window algorithm

Today = **fixed window** (`InMemoryRateLimiter`, `rate-limit.ts:46-72`). Recommendation: keep behavior equivalent with **`@upstash/ratelimit` fixed window** so the observable semantics (limit N per `WINDOW_MS`, `Retry-After` = seconds to window end) match exactly and no test assertions shift. Sliding-window is available and slightly smoother against burst-at-boundary, but changing the algorithm changes observable counts → defer it; not needed to hit M6.2's goal. Decision flagged in §9.

### Preserving env + bucket config exactly

The Redis limiter MUST read the same envs and defaults as `limiterFor` (`rate-limit.ts:102-120`):

| Bucket | Env | Default |
|---|---|---|
| `write` | `RATE_LIMIT_WRITE_MAX` | 30 |
| `agentsRead` | `RATE_LIMIT_AGENTS_MAX` | 30 |
| `signalsRead` | `RATE_LIMIT_SIGNALS_MAX` | 60 |
| `providerRead` | `RATE_LIMIT_READ_MAX` | 60 |
| (all) window | `RATE_LIMIT_WINDOW_MS` | 60000 |
| kill-switch | `RATE_LIMIT_DISABLED=1` → Noop | — |

**Finding (env gap to fix in M6.2):** `signalsRead` reads `RATE_LIMIT_SIGNALS_MAX` (`rate-limit.ts:112`) but that var is **absent from both** `lib/env.ts` `optionalSchema` (`env.ts:65-70` lists only WRITE/READ/AGENTS) **and** `.env.example` (lines 69-77). It works via `envInt` fallback (60) but is undocumented and unvalidated. Add `RATE_LIMIT_SIGNALS_MAX` to `optionalSchema` and `.env.example` as part of this milestone (backend-developer + devops-engineer).

### The one real call-site cost: sync → async

`enforceRateLimit(userId, bucket)` is **synchronous today** — it returns `void` and throws `TooManyRequests` (`rate-limit.ts:127-132`). Route handlers call it bare: `enforceRateLimit(userId, "providerRead")` (e.g. `app/api/company-status/route.ts:38`, `app/api/agents/route.ts:18`, and ~15 more per the call-site sweep).

A network-backed store **cannot** be awaited inside a synchronous function. Therefore, honestly:

> **You cannot keep the exact synchronous signature.** `enforceRateLimit` must become `async` → `Promise<void>`, and every call-site must add `await`.

This is the **only** required call-site change, and it is mechanical: `enforceRateLimit(...)` → `await enforceRateLimit(...)`. All call-sites are already inside `async` route handlers, so no structural change is needed — just the `await` keyword. Do NOT attempt to hide the network I/O behind a fake-sync facade (`deasync`, blocking loops); that is worse than the honest `await`.

Everything else is preserved: bucket names, the `` `${bucket}:${userId}` `` keying (tenant-keyed, never IP — `rate-limit.ts:122-128`), the `TooManyRequests` → 429 + `Retry-After` mapping, the `RATE_LIMIT_DISABLED` kill-switch, and the per-bucket maxes. The `RateLimitResult` shape (`rate-limit.ts:15-21`) is reused unchanged.

**Caches are already async** — `getAgentsCached` returns `Promise` (`agents-cache.ts:38`) and `getAssetPrice`/`getFxUsdThb` are async (`service.ts:54,82`). So cache migration needs **zero signature change**; only internals swap from Map lookups to `cache.get/set`.

## 5. Secret / provisioning plan & fail-mode policy

### New env vars

| Var | Required? | Purpose |
|---|---|---|
| `UPSTASH_REDIS_REST_URL` | Optional at schema level, required for distributed mode | Upstash REST endpoint (HTTPS). |
| `UPSTASH_REDIS_REST_TOKEN` | Optional at schema level, required for distributed mode | Bearer token for the REST API. Secret — never logged (follow `redactSecrets` / `verify-secret.ts` house rule). |
| `STORE_DRIVER` (optional) | Optional | `"memory" \| "redis"`. Default: `redis` when the Upstash pair is present, else `memory`. Lets local/CI force in-memory without creds. |

House-style wiring (devops-engineer):
- Add all three to `.env.example` in a new "Distributed store (Upstash Redis)" block, mirroring the annotated style of the existing rate-limit block (`.env.example:62-85`).
- Add to `lib/env.ts` `optionalSchema` (`env.ts:54-71`) as `.optional()` strings — they must **not** be in `requiredSchema`, to preserve the keyless build/boot contract (`env.ts:5-16`). The store selection degrades to in-memory when absent, exactly like providers degrade to mock.
- Vercel env: documented as a plan only. **Standing hold: no provisioning/deploy is performed by this milestone.** Implementation-time devops task, not now.

### Fail-mode policy (when Redis is unreachable or creds missing)

Two different postures, deliberately, because the two subsystems have different risk profiles:

**(a) Cache → FAIL-OPEN (degrade to source of truth).**
On a Redis error/timeout, `cache.get` returns `null` (treated as a miss) and `cache.set` is best-effort/no-throw. The caller then does what it already does on a cache miss: fetch live, or fall back to snapshot/mock. This exactly matches the existing never-throw fallback philosophy in `service.ts:54-79` ("Never throws: always resolves"). Cost: a Redis outage means more provider calls (the shield weakens) — acceptable and self-healing. A cache is an optimization; it must never take the request path down.

**(b) Rate limiter → FAIL-CLOSED on missing creds, FAIL-OPEN (with alarm) on transient outage. Recommended posture:**

- **Creds missing / misconfigured in production:** fail toward the **in-memory limiter**, NOT toward "no limit." Rationale: the in-memory limiter is a real (if per-instance) shield; silently running with *no* limiter would remove a security control. This mirrors the fail-closed instinct of `verifySharedSecret` (`verify-secret.ts:33`) and `authorizeCron` (`snapshot-batch.ts:51`) — a missing secret must never widen access. In `STORE_DRIVER=memory` / dev, in-memory is the normal path anyway.
- **Transient Redis outage at request time (creds present, call fails):** here is the genuine security-vs-availability tradeoff. Options:
  - *Fail-open* (allow the request): a Redis blip can't 500 every API call, but during the blip the limit is not enforced.
  - *Fail-closed* (429 everything): preserves the cap but converts a cache-tier outage into a total API outage for all users — disproportionate.
  - **Recommendation: fail-open to the in-memory limiter fallback, and emit a `console.error` (redacted) so the outage is observable.** This keeps a best-effort per-instance cap during the blip (better than fully open), keeps the app up, and self-heals when Redis returns. Do NOT fail fully open (no counting at all).

This is a security-relevant call; the final decision is escalated to the CEO in §9 (Open Question O-3). The recommendation above (in-memory fallback, never fully open, always log) is the safe default if no other guidance is given.

## 6. Migration / rollout plan (phased, backwards-compatible)

The strategy: introduce the abstraction **behind the current public functions**, with two adapters selected at runtime. Tests and local dev use the in-memory adapter; prod uses Redis. No behavior change is shipped until each phase is green.

**Phase 0 — Contract lock (this doc).** Interfaces in §3, key schema, fail-mode policy. CEO review gate.

**Phase 1 — Abstraction + in-memory adapter (backend-developer).**
- Add `lib/store/types.ts` (interfaces), `lib/store/memory-adapter.ts` (wraps the *existing* `InMemoryRateLimiter` logic and a `Map`-based cache — literally the current code moved behind the interface), and `lib/store/index.ts` (factory that picks the driver from `STORE_DRIVER` / presence of Upstash creds, defaulting to memory).
- Refactor `enforceRateLimit` to `async` and delegate to the selected `DistributedRateLimiter`. Update the ~17 call-sites to `await` (mechanical). Keep `__resetRateLimiters` (`rate-limit.ts:135`) working against the memory adapter for tests.
- No Redis yet → this phase changes *structure only*, behavior identical. Full test suite must stay green here (proves the refactor is behavior-preserving).

**Phase 2 — Redis adapter (backend-developer).**
- Add `lib/store/redis-adapter.ts`: `@upstash/ratelimit` for the limiter, `@upstash/redis` `get/set(ex)/del` for the cache. This is the ONLY file that imports the vendor SDK (portability boundary from §2). Package install happens in this phase, not now.
- Wire cache internals of `agents-cache.ts` and `market-data/service.ts` to the `DistributedCache` (their public async signatures are unchanged — §4).

**Phase 3 — Env, docs, provisioning plan (devops-engineer).**
- `.env.example` block, `lib/env.ts` `optionalSchema` additions (incl. the `RATE_LIMIT_SIGNALS_MAX` gap fix), `scripts/check-env.mjs` awareness if applicable. Provision Upstash DB in `iad1`/`us-east-1` and set Vercel envs — **executed later, on CEO deploy authorization, not in this milestone.**

**Phase 4 — QA + rollout (qa-engineer).** Unit tests via memory adapter (below), optional integration test against a real Upstash DB behind a flag. Roll out with `STORE_DRIVER=redis` in prod; instant rollback = set `STORE_DRIVER=memory` (no code change).

Backwards-compat guarantees: cache call-sites unchanged; limiter call-sites change only by adding `await`; kill-switch and env maxes preserved; memory adapter keeps CI/local working with no Redis.

## 7. Test-strategy hooks

- **In-memory adapter = the test seam.** Every limiter/cache unit test runs against `memory-adapter` (set `STORE_DRIVER=memory` or inject the adapter), so no live Redis is needed in CI. `__resetRateLimiters` (`rate-limit.ts:135`) and `__resetAgentsCache` (`agents-cache.ts:50`) remain the reset seams, now clearing the memory adapter.
- **Async assertions:** tests must `await enforceRateLimit(...)` after the sync→async change; the 429-after-N-hits and `Retry-After` assertions transfer unchanged since the fixed-window semantics are preserved (§4).
- **Additive R-QA tests folded into M6.2/M6.3:**
  - **R-QA-01** — `verifySharedSecret` fail-closed coverage (`verify-secret.ts:29-36`): unset-secret → false, mismatch → false, match → true. Independent of the store, lands alongside this work.
  - **R-QA-02** — `signalsRead` limiter coverage: N `signalsRead` hits allowed then 429; confirms `RATE_LIMIT_SIGNALS_MAX` (default 60) is honored (ties into the env-gap fix, §4).
  - **R-QA-03** — TradingView webhook POST fail-closed: unauthenticated/secret-missing POST rejected (webhook secret fail-closed, per the M6.1 commit line). Store-independent; travels with M6.2/M6.3.
- No test may depend on a live Upstash instance to pass; the Redis adapter's own integration test is opt-in behind an env flag so the default `npm test` stays hermetic.

## 8. Named tradeoffs & risks

- **REST-per-call latency.** Every limiter check and cache read is now an HTTPS round trip (~single-to-low-tens ms in-region) vs. a ~0ms Map lookup. On the hottest path (TV webhook GET polled every 10s, `MissionControlClient.tsx:69`) this is one extra round trip per poll — acceptable, but real. Mitigation: co-locate Upstash in `iad1`; use `@upstash/ratelimit` (one command, not multiple round trips).
- **Free-tier command volume.** Rough per-open-dashboard-tab volume (rate-limited GETs): TV 6/min + crypto 1.33 + company 1.33 + agents 2 + portfolios 1.33 + affiliate 1 ≈ **~13 rate-limited requests/min**. At ~1–2 Upstash commands per limiter check (plus cache get/set on price/FX/agents), call it ~20–30 commands/min/tab ≈ **~1.5–2k commands/hour/tab**. An 8-hour active day with one tab ≈ **~12–16k commands/day**. Upstash's free tier (order ~500k commands/month) comfortably covers a **single-user personal-scale** deployment, but a few always-open tabs, or leaving the dashboard open 24/7, will approach it. → Open Question O-1.
- **Vendor lock-in vs. portability.** `@upstash/redis`/`@upstash/ratelimit` are Upstash-specific SDKs, but the commands are standard Redis and are confined to `redis-adapter.ts`. Swapping to any Redis-compatible host = rewrite one adapter file. Accepted, low risk.
- **Local-dev story.** Devs without Upstash creds run `STORE_DRIVER=memory` (or just omit creds → auto memory) and get today's behavior. No docker-redis dependency imposed. This is the deliberate reason for the two-adapter design.
- **Sync→async blast radius.** ~17 call-sites gain `await`. Low risk (mechanical, all already async contexts) but touches many files — do it as one reviewable Phase-1 commit, not sprinkled.
- **Fixed-window boundary burst.** Inherited from today's algorithm (unchanged). A user can send 2×limit across a window boundary. Known, acceptable; sliding-window is the future upgrade if it ever matters.
